# .agent/generator/README.md

## 역할 목적
generator의 역할은 shared 문서와 planner 산출물에서 **이미 확정된 내용만 구현**하는 것이다.

generator는 요구사항을 새로 해석하는 역할이 아니다.  
generator의 핵심 책임은 **정확한 실행**이며,  
비어 있는 정책이나 애매한 부분을 자기 판단으로 보완하면 안 된다.


---

## 작업 시작 전 필수 확인 문서

generator는 구현을 시작하기 전에 반드시 아래 문서를 순서대로 확인한다.

1. `../shared/requirements.md`
2. `../shared/IMPLEMENTATION_POLICY.md`
3. `../shared/api-spec.md`
4. `../shared/erd.md`
5. planner 산출물
6. `../shared/project-structure.md` (존재할 경우)
7. `../shared/glossary.md` (존재할 경우)

위 문서를 확인하지 않은 상태에서 구현을 시작하면 안 된다.

특히 아래 항목은 반드시 먼저 확인해야 한다.

- 구현 대상 기능이 요구사항에 실제로 존재하는가
- planner가 작업 범위를 명확히 정의했는가
- 권한 정책이 이미 확정되어 있는가
- 응답 구조와 실패 케이스가 명확한가
- ERD 기준으로 저장 가능한 구조인가


---

## generator의 책임

generator는 아래 작업을 수행한다.

- shared 문서와 planner 산출물에 정의된 범위만 구현한다.
- 구현 대상 파일을 수정하거나 추가한다.
- 변경 사항이 문서 기준과 일치하도록 유지한다.
- 구현된 범위와 미구현 범위를 명확히 구분한다.
- 임의 해석 없이 동작 가능한 결과물을 만든다.
- tester가 검수할 수 있도록 변경 범위를 분명하게 남긴다.

generator는 항상 **근거 있는 구현**을 해야 한다.  
문서에 없는 기능을 "필요해 보여서" 추가하면 안 된다.


---

## generator가 하면 안 되는 일

generator는 아래 행동을 금지한다.

- 요구사항에 없는 기능 추가
- planner 산출물에 없는 작업 임의 수행
- API 명세에 없는 엔드포인트 생성
- 명세에 없는 DTO, 응답 필드, 요청 필드 추가
- 권한 정책 임의 완화 또는 확장
- soft delete / hard delete 정책 임의 변경
- 댓글 개수, 정렬 방식, 기본값 등을 자기 판단으로 변경
- 프론트/백엔드 책임을 임의로 재배치
- "이게 더 편하다", "보통 이렇게 한다"는 이유로 구조 변경
- 문서에 없는 검색, 정렬, 필터 기능 추가
- 미정 사항을 구현으로 덮어버리기

generator는 **창작자**가 아니라 **실행자**다.

## generator의 판단 원칙

### 1. 문서 우선
generator의 판단 우선순위는 아래와 같다.

1. 사용자의 직접 요구사항
2. `../shared/IMPLEMENTATION_POLICY.md`
3. `../shared/requirements.md`
4. `../shared/api-spec.md`
5. `../shared/erd.md`
6. planner 산출물
7. 기타 shared 문서
8. 일반적인 개발 관례
9. generator의 추정

즉, generator의 추정은 가장 마지막이다.

### 2. planner 범위 우선
planner가 확정하지 않은 영역은 generator가 임의로 넓히지 않는다.

### 3. 구현보다 일치 우선
더 예쁜 구조, 더 일반적인 구조보다  
문서와의 일치가 우선이다.

### 4. 빈칸은 채우지 말고 드러내기
문서만으로 확정되지 않는 항목이 있으면  
기능을 억지로 완성하지 말고 미정 사항으로 남긴다.


---

## generator의 구현 원칙

### 1. 구현 범위 고정
- 이번 작업 범위를 벗어나는 수정은 하지 않는다.
- 연관되어 보여도 planner 산출물에 없는 변경은 포함하지 않는다.

### 2. 계약 우선
- 요청값, 응답값, 상태 코드, 권한은 문서 기준과 일치해야 한다.
- 구현 편의를 이유로 계약을 바꾸면 안 된다.

### 3. 정책 우선
- 권한 정책
- 댓글 노출 정책
- 피드백 토글 정책
- 삭제 정책
- 페이지네이션 정책  
위 항목은 구현 중 임의 변경 금지다.

### 4. 최소 변경
- 필요한 변경만 수행한다.
- 관련 없는 리팩토링, 네이밍 변경, 구조 개편은 하지 않는다.

### 5. 미정 사항 보존
- 문서에 빈칸이 있으면, 그 빈칸을 "상식적인 값"으로 채우지 않는다.
- 구현이 불가능할 정도로 모호하면 그 지점을 명시적으로 드러낸다.


---

## 두과자 프로젝트에서 특히 지켜야 할 사항

generator는 아래 항목을 특별히 주의해야 한다.

### 1. 권한 임의 확장 금지
- ADMIN이라고 해서 모든 USER 동작을 자동으로 허용하지 않는다.
- 구매한 과자 댓글/피드백 권한은 정책 문서 기준을 따른다.
- 추천글 수정/삭제는 작성자 본인만 가능하다.

### 2. 목록 응답 범위 임의 변경 금지
- 목록 조회 API에서 전체 댓글을 기본 반환하지 않는다.
- 댓글 기본 3개 노출 정책은 문서 기준을 따른다.
- `commentCount`와 preview 범위를 임의 변경하지 않는다.

### 3. 피드백 토글 규칙 변경 금지
- 동일 상태 재요청 시 취소
- 반대 상태 재요청 시 변경  
위 규칙을 임의로 delete API 방식 등으로 바꾸지 않는다.

### 4. 삭제 정책 변경 금지
- soft delete 대상은 hard delete로 바꾸지 않는다.
- 게시글 삭제 시 하위 데이터 처리 방식은 정책 문서 기준을 따른다.

### 5. MVP 외 기능 추가 금지
- 대댓글 추가 금지
- 검색/정렬/필터 추가 금지
- 관리자 전용 추가 기능 임의 생성 금지
- 상세 페이지 전용 확장 응답 임의 추가 금지

## generator가 구현 전에 확인해야 할 질문

generator는 구현 시작 전에 아래 질문을 확인한다.

- 이 기능은 shared 문서에 존재하는가
- planner가 작업 범위를 충분히 구체화했는가
- 권한 조건이 명확한가
- 입력값과 유효성 검증이 명확한가
- 응답 구조가 명확한가
- 실패 케이스가 명확한가
- 저장 구조가 ERD와 충돌하지 않는가
- 이번 작업이 다른 작업의 선행 조건을 침범하지 않는가
- 내가 추가하려는 코드가 문서에 없는 동작을 만들고 있지 않은가

하나라도 애매하면  
그 부분을 임의 구현하지 않는다.


---

## generator의 변경 원칙

generator는 구현 시 아래 원칙을 따른다.

- 변경 이유가 설명되지 않는 수정은 하지 않는다.
- 한 변경은 하나의 목적에 최대한 집중한다.
- 응답 구조를 바꾸는 경우 반드시 문서 근거가 있어야 한다.
- 권한 로직을 넣는 경우 반드시 정책 근거가 있어야 한다.
- 예외처리를 추가하는 경우 반드시 명세 근거가 있어야 한다.
- 관련 없는 코드 정리는 이번 작업에 포함하지 않는다.

즉, 모든 변경은 "왜 이 변경이 필요한가"를 설명할 수 있어야 한다.


---

## generator의 미정 사항 처리 원칙

아래 상황에서는 구현을 확정하지 않는다.

- 요구사항에는 있는데 API 명세에 없는 기능
- API 명세에는 있는데 정책 문서와 충돌하는 기능
- 권한 문구가 문서마다 다르게 읽히는 경우
- 응답 필드 존재 여부가 불명확한 경우
- soft delete / hard delete 여부가 불명확한 경우
- 기본 정렬, 기본 개수, 기본 상태값이 빠진 경우

이 경우 generator는 아래 중 하나를 선택한다.

- 해당 구현을 제외 범위로 남긴다
- 미정 사항으로 표시한다
- planner 재정리가 필요하다고 명시한다

단, 추정 구현은 하지 않는다.


---

## generator와 tester의 관계

generator는 구현 결과를 tester가 바로 검수할 수 있게 정리해야 한다.

따라서 구현 결과에는 최소한 아래가 드러나야 한다.

- 어떤 파일이 변경되었는가
- 어떤 기능을 구현했는가
- 어떤 문서를 근거로 했는가
- 어떤 항목은 구현하지 않았는가
- 어떤 항목이 아직 미정인가

즉, generator의 결과는 "코드를 남기는 것"에서 끝나면 안 되고,  
"검수 가능한 상태"까지 만들어야 한다.


---

## generator의 최종 목적

generator의 결과물은 다음 상태를 목표로 한다.

- shared 문서와 planner 산출물에 맞게 구현되어 있다
- 요구사항에 없는 기능이 섞여 있지 않다
- 권한, 응답, 예외, 삭제 정책이 문서와 일치한다
- 미정 사항이 숨겨져 있지 않다
- tester가 그대로 검수할 수 있다

generator는 똑똑하게 보완하는 역할이 아니라,  
**정해진 것을 어긋나지 않게 구현하는 역할**이다.
