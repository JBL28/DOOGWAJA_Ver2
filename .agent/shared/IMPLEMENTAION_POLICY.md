# IMPLEMENTATION_POLICY.md

## 목적
이 문서는 두과자 프로젝트 구현 시, 요구사항과 API 명세 사이에서 발생할 수 있는 해석 차이를 줄이고,  
에이전트 또는 개발자가 임의로 정책을 보완하지 않도록 하기 위한 기준 문서이다.

본 문서의 우선순위는 다음과 같다.

1. 요구사항 문서
2. IMPLEMENTATION_POLICY.md
3. API 명세서
4. 구현체 내부 판단

즉, API 명세에 비어 있거나 애매한 부분이 있더라도, 이 문서에 명시된 정책을 우선 적용한다. :contentReference[oaicite:0]{index=0} :contentReference[oaicite:1]{index=1}

## 1. 공통 원칙

### 1-1. 임의 확장 금지
- 명시되지 않은 기능은 구현하지 않는다.
- 편의상 추가 가능한 기능이라도, 요구사항/명세/정책 문서에 없으면 임의 추가하지 않는다.
- 특히 아래 항목은 임의 판단을 금지한다.
  - 권한 확장
  - 응답 필드 추가/생략
  - soft delete / hard delete 방식 변경
  - 댓글/피드백 토글 규칙 변경
  - 목록 조회 시 포함 데이터 범위 변경

### 1-2. MVP 우선
- MVP 범위를 벗어나는 기능은 구현하지 않는다.
- 대댓글 기능은 제외한다. :contentReference[oaicite:2]{index=2}

### 1-3. 화면 요구사항을 API에서 보조해야 한다
- 프론트엔드가 최종 렌더링을 담당하더라도,
- API는 화면 요구사항을 안정적으로 구현할 수 있도록 필요한 최소 구조를 제공해야 한다.
- 프론트에서 억지로 자르거나 조합해야만 동작하는 구조는 지양한다.

## 2. 권한 정책

## 2-1. 역할 정의
- 역할은 `USER`, `ADMIN` 두 가지다.
- `ADMIN`은 관리 기능 수행을 위한 역할이며, 자동으로 모든 USER 행동 권한을 포함한다고 간주하지 않는다.
- 즉, **엔드포인트별 허용 역할을 명시적으로 따르며**, "관리자니까 다 된다"로 해석하지 않는다. :contentReference[oaicite:3]{index=3}

## 2-2. 권한 매트릭스

| 기능 | USER | ADMIN |
|---|---:|---:|
| 로그인/로그아웃/토큰 재발급 | O | O |
| 내 정보 조회/수정/탈퇴 | O | O |
| 추천글 조회 | O | O |
| 추천글 작성 | O | X |
| 추천글 수정/삭제(본인 글) | O | X |
| 추천글 댓글 작성 | O | O |
| 추천글 댓글 수정/삭제(본인 댓글) | O | O |
| 추천글 피드백 | O | O |
| 추천글 댓글 피드백 | O | O |
| 구매한 과자 조회 | O | O |
| 구매한 과자 등록/수정/삭제 | X | O |
| 구매한 과자 상태 변경 | O | O |
| 구매한 과자 댓글 작성 | O | X |
| 구매한 과자 댓글 수정/삭제(본인 댓글) | O | X |
| 구매한 과자 피드백 | O | X |
| 구매한 과자 댓글 피드백 | O | X |
| 유저 목록 조회/상태 변경/삭제 | X | O |

## 2-3. 권한 해석 상세
- 추천글 관련 기능은 로그인 사용자라면 ADMIN도 사용할 수 있다.
- 구매한 과자 관련 댓글/피드백은 요구사항 문구상 "관리자가 아닌 사용자"가 수행한다.
- 따라서 ADMIN은 구매한 과자에 대해 댓글 작성 및 좋아요/싫어요를 할 수 없다. :contentReference[oaicite:4]{index=4}
- 구매한 과자 상태 변경은 모두 가능하다. USER와 ADMIN 모두 허용한다. :contentReference[oaicite:5]{index=5}

## 2-4. 본인 작성물 정책
- 추천글 수정/삭제는 작성자 본인만 가능하다.
- 댓글 수정/삭제는 작성자 본인만 가능하다.
- ADMIN이라고 해서 타인의 추천글/댓글을 임의 수정/삭제하지 않는다.
- 관리 목적의 삭제 기능이 필요하다면 별도 관리자 엔드포인트를 추가하기 전까지 구현하지 않는다.

## 3. 인증 / 계정 정책

## 3-1. 회원가입 유효성
- `loginId`는 5자 이상이어야 한다.
- `password`는 영문과 숫자를 각각 1자 이상 포함해야 한다.
- `nickname`은 필수값이다. :contentReference[oaicite:6]{index=6} :contentReference[oaicite:7]{index=7}

## 3-2. 추가 유효성 정책
- `loginId`, `nickname`, 게시글 제목, 댓글 내용은 공백만 입력된 값을 허용하지 않는다.
- 앞뒤 공백은 저장 전에 trim 처리한다.
- trim 후 빈 문자열이면 유효성 실패로 처리한다.

## 3-3. 계정 상태 정책
- `ACTIVATED`: 정상 사용 가능
- `DEACTIVATED`: 로그인 불가
- `DELETED`: 로그인 불가, 탈퇴 계정
- 회원 탈퇴와 관리자 삭제는 모두 soft delete 정책을 따른다. 즉, 실제 row 삭제 대신 상태값을 변경한다. :contentReference[oaicite:8]{index=8} :contentReference[oaicite:9]{index=9}

## 4. 추천글 정책

## 4-1. 생성/수정
- 추천글은 `name`, `reason` 두 필드를 가진다.
- 둘 중 하나라도 비어 있으면 생성/수정할 수 없다. :contentReference[oaicite:10]{index=10}

## 4-2. 삭제
- 추천글 삭제 시 해당 추천글에 연결된 아래 데이터는 함께 삭제한다.
  - 추천 댓글
  - 추천글 피드백
  - 추천 댓글 피드백
- 이 삭제는 사용자 입장에서 "완전히 삭제된 것처럼" 동작해야 한다.
- 구현 방식은 cascade delete 또는 서비스 단 수동 삭제 중 택할 수 있으나, 외부 동작은 동일해야 한다.

## 4-3. 조회
- 추천글 목록 조회는 페이지네이션을 적용한다.
- 기본 page size는 10이다. :contentReference[oaicite:11]{index=11} :contentReference[oaicite:12]{index=12}

## 5. 구매한 과자 정책

## 5-1. CRUD
- 구매한 과자 등록/수정/삭제는 ADMIN만 수행한다. :contentReference[oaicite:13]{index=13}
- 등록 시 기본 상태값은 `SHIPPING`이다. :contentReference[oaicite:14]{index=14}

## 5-2. 상태 변경
- 구매한 과자의 상태값은 아래 3개만 허용한다.
  - `SHIPPING`
  - `IN_STOCK`
  - `OUT_OF_STOCK`
- USER와 ADMIN 모두 상태를 변경할 수 있다. :contentReference[oaicite:15]{index=15} :contentReference[oaicite:16]{index=16}
- 같은 상태값으로 다시 요청해도 에러로 처리하지 않는다.
- 같은 상태 재요청은 `200 OK`와 현재 상태를 그대로 반환한다.
- 상태 변경 이력 저장 기능은 MVP에서 구현하지 않는다.
- 마지막 변경자 저장 기능도 MVP에서 구현하지 않는다.

## 5-3. 삭제
- 구매한 과자 삭제 시 해당 과자에 연결된 아래 데이터는 함께 삭제한다.
  - 구매 과자 댓글
  - 구매 과자 피드백
  - 구매 과자 댓글 피드백

## 6. 댓글 정책

## 6-1. 공통
- 댓글은 추천글과 구매한 과자 게시글에만 작성할 수 있다.
- 대댓글 기능은 구현하지 않는다. :contentReference[oaicite:17]{index=17}
- 댓글 내용은 비어 있을 수 없다.
- 공백만 있는 댓글은 허용하지 않는다.

## 6-2. 댓글 목록 노출 정책
- 댓글은 기본적으로 **3개만 노출**한다.
- 댓글 수가 3개를 초과하면 프론트에서 펼치기/접기를 제공한다.
- 접힌 상태에서는 최근 댓글 3개만 보여준다.
- 펼친 상태에서는 전체 댓글을 보여준다. :contentReference[oaicite:18]{index=18} :contentReference[oaicite:19]{index=19}

## 6-3. API 응답 정책
- 목록 조회 API는 댓글 전체를 반환하지 않는다.
- 목록 조회 API는 각 게시글에 대해 아래만 반환한다.
  - `commentCount`
  - `commentPreview` (최대 3개)
- 상세 조회 API 또는 별도 댓글 조회 API에서 전체 댓글을 반환한다.

## 6-4. 댓글 정렬 정책
- `commentPreview`는 최신순 기준 최근 3개를 반환한다.
- 전체 댓글 조회 역시 기본 정렬은 최신순으로 한다.
- 프론트는 서버가 반환한 순서를 그대로 사용한다.

## 6-5. 댓글 수정/삭제
- 댓글 수정/삭제는 작성자 본인만 가능하다.
- ADMIN이더라도 구매 과자 댓글은 수정/삭제할 수 없다.
- ADMIN이 추천 댓글을 작성한 경우, 본인 댓글은 수정/삭제할 수 있다.

## 7. 피드백 정책

## 7-1. 공통 규칙
- 피드백 대상:
  - 추천글
  - 추천 댓글
  - 구매 과자
  - 구매 과자 댓글
- 피드백 값:
  - `LIKE`
  - `DISLIKE`
- 한 사용자는 한 대상에 대해 좋아요/싫어요 중 하나만 가질 수 있다. :contentReference[oaicite:20]{index=20} :contentReference[oaicite:21]{index=21}

## 7-2. 토글 규칙
모든 피드백 API는 동일한 토글 규칙을 따른다.

| 기존 상태 | 요청 상태 | 결과 |
|---|---|---|
| 없음 | LIKE | LIKE 생성 |
| 없음 | DISLIKE | DISLIKE 생성 |
| LIKE | LIKE | 삭제 |
| DISLIKE | DISLIKE | 삭제 |
| LIKE | DISLIKE | DISLIKE로 변경 |
| DISLIKE | LIKE | LIKE로 변경 |

- 즉, 동일한 값을 다시 보내면 취소로 처리한다.
- 다른 값을 보내면 반대 상태로 변경한다. :contentReference[oaicite:22]{index=22} :contentReference[oaicite:23]{index=23}

## 7-3. 응답 규칙
피드백 반영 후 응답은 항상 아래 필드를 포함한다.
- `myFeedback`
- `likeCount`
- `dislikeCount`

## 7-4. 구매 과자 피드백 권한
- 구매 과자 게시글 및 구매 과자 댓글 피드백은 USER만 가능하다.
- ADMIN은 구매 과자에 대한 피드백을 남길 수 없다. :contentReference[oaicite:24]{index=24}

## 8. 목록 / 상세 응답 계약

## 8-1. 목록 조회 API 목적
목록 조회 API는 메인 화면 렌더링에 필요한 최소 데이터를 안정적으로 제공하는 것을 목적으로 한다.

## 8-2. 목록 조회 API 포함 필드
추천글 목록과 구매 과자 목록은 각각 아래 정보를 포함해야 한다.

- 게시글 기본 정보
- 작성자 정보(추천글인 경우)
- 상태 정보(구매 과자인 경우)
- 피드백 요약
  - `likeCount`
  - `dislikeCount`
  - `myFeedback`
- 댓글 수
  - `commentCount`
- 댓글 미리보기
  - `commentPreview` (최대 3개)

## 8-3. 목록 조회 API 제외 필드
다음은 목록 조회 API에서 기본적으로 포함하지 않는다.
- 전체 댓글 배열
- 댓글의 댓글
- 상세 화면 전용 확장 정보

## 8-4. 상세 조회 API
상세 조회 API는 전체 댓글을 포함할 수 있다.
다만 MVP 범위에서 별도 상세 페이지가 없다면, 메인 화면에서 필요한 수준을 넘는 과도한 중첩 데이터는 반환하지 않는다.

## 8-5. 페이지네이션
- 메인 페이지 목록은 10개씩 페이지네이션한다. :contentReference[oaicite:25]{index=25}
- page는 0부터 시작한다.
- size 기본값은 10이다.

## 9. 에러 처리 정책

## 9-1. 공통 원칙
- 의미에 맞는 HTTP 상태 코드를 사용한다.
- 에러 메시지는 프론트에서 바로 표시해도 이해 가능한 수준으로 작성한다. :contentReference[oaicite:26]{index=26}

## 9-2. 상태 코드 기준
- `400 Bad Request`
  - 유효성 검증 실패
  - 잘못된 요청값
  - 현재 비밀번호 불일치
- `401 Unauthorized`
  - 인증 실패
  - 토큰 없음
  - 토큰 만료
  - 유효하지 않은 토큰
- `403 Forbidden`
  - 권한 없음
  - 비활성화 계정
  - 탈퇴 계정
- `404 Not Found`
  - 대상 리소스를 찾을 수 없음
- `409 Conflict`
  - 중복 아이디 등 고유값 충돌
- `500 Internal Server Error`
  - 서버 내부 오류

## 9-3. 대표 실패 케이스
- 없는 추천글 수정/삭제 요청: `404`
- 타인의 추천글 수정/삭제 요청: `403`
- 없는 댓글 수정/삭제 요청: `404`
- 타인의 댓글 수정/삭제 요청: `403`
- 비활성화 계정 로그인: `403`
- 탈퇴 계정 로그인: `403`
- 잘못된 상태값 요청: `400`

## 10. 삭제 정책

## 10-1. 유저 탈퇴 / 관리자 삭제
- 유저 탈퇴와 관리자 삭제는 모두 soft delete다.
- 즉, `user.status = DELETED` 처리한다. :contentReference[oaicite:27]{index=27} :contentReference[oaicite:28]{index=28}

## 10-2. 탈퇴 유저의 기존 데이터 처리
- MVP에서는 탈퇴 유저의 기존 추천글/댓글/피드백을 자동 삭제하지 않는다.
- 이미 작성된 데이터는 유지한다.
- 단, 이후 로그인 및 신규 활동은 불가능하다.

## 10-3. 표시 정책
- 탈퇴 유저의 닉네임 표시 정책은 MVP에서는 기존 저장 닉네임을 그대로 사용한다.
- 익명화 기능은 MVP 범위에서 구현하지 않는다.

## 10-4. 게시글 삭제
- 게시글 삭제는 하위 댓글 및 피드백까지 함께 제거한다.
- 사용자 관점에서 orphan 데이터가 남아 보이면 안 된다.


---

## 11. 구현 금지 사항

다음은 에이전트 또는 개발자가 임의로 구현하면 안 된다.

- 댓글 전체 반환을 목록 API 기본값으로 두는 것
- ADMIN에게 구매 과자 댓글/피드백 권한을 주는 것
- 대댓글 기능 추가
- 추천글을 ADMIN도 작성 가능하게 바꾸는 것
- soft delete를 hard delete로 바꾸는 것
- 피드백 토글 규칙을 별도 delete API 방식으로 임의 변경하는 것
- 메인 화면 페이지 크기를 10 외 값으로 고정하는 것
- 요구사항에 없는 검색/정렬/필터 기능 추가

## 12. 최종 기준 요약

이 프로젝트의 구현 기준은 아래와 같다.

- USER는 추천글 중심의 참여와 구매 과자에 대한 반응을 담당한다.
- ADMIN은 구매 과자 관리와 회원 관리만 담당한다.
- 구매 과자의 상태는 모두가 바꿀 수 있다.
- 댓글은 기본 3개만 보여주며, 목록 API는 그 구조를 직접 지원해야 한다.
- 좋아요/싫어요는 모든 대상에서 동일한 토글 규칙을 따른다.
- 삭제 정책과 권한 정책은 문서 기준으로 고정하며, 임의 확장하지 않는다.
