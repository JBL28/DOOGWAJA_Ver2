# .agent/README.md

## 목적
이 폴더는 에이전트가 두과자 프로젝트를 임의로 해석하지 않고,  
정해진 요구사항과 정책을 기준으로 계획·구현·검수를 수행하도록 하기 위한 작업 지침 폴더이다.

이 폴더의 목적은 다음과 같다.

- 공통 기준 문서를 한곳에서 참조하도록 강제한다.
- 역할별 책임을 분리한다.
- 요구사항에 없는 기능을 임의로 추가하지 못하게 한다.
- 구현보다 해석 통제를 우선한다.


---

## 가장 먼저 해야 할 일

이 폴더를 사용하는 에이전트는 작업 시작 전에 반드시 아래 순서로 문서를 확인한다.

1. `shared/requirements.md`
2. `shared/IMPLEMENTATION_POLICY.md`
3. `shared/api-spec.md`
4. `shared/erd.md`
5. 자신의 역할 폴더 문서

즉, 역할 문서를 먼저 읽는 것이 아니라,  
반드시 `shared/`의 공통 기준을 먼저 읽고 그 다음에 역할 문서를 읽는다.

`shared/`를 확인하지 않은 상태에서 작업을 시작하면 안 된다.


---

## shared 폴더 사용 원칙

`shared/`는 planner, generator, tester가 공통으로 따라야 하는 기준 문서 모음이다.

여기에는 아래와 같은 문서가 위치한다.

- 요구사항 문서
- 구현 정책 문서
- API 명세서
- ERD 문서
- 프로젝트 구조 문서
- 용어집

### shared 문서의 성격
`shared/`의 문서는 프로젝트의 **사실**, **정책**, **계약**, **제약**을 정의한다.

즉, 아래 내용은 반드시 `shared/` 기준을 따른다.

- 어떤 기능을 구현해야 하는가
- 누가 어떤 기능을 사용할 수 있는가
- 어떤 응답 형식을 반환해야 하는가
- 어떤 동작을 금지하는가
- 삭제, 토글, 예외 처리 정책이 무엇인가

### shared 문서 우선순위
문서 충돌 시 우선순위는 아래와 같다.

1. 사용자의 직접 요구사항
2. `shared/IMPLEMENTATION_POLICY.md`
3. `shared/requirements.md`
4. `shared/api-spec.md`
5. `shared/erd.md`
6. 각 역할 문서
7. 에이전트의 일반적 추론

즉, 에이전트의 추론은 가장 마지막이다.  
명시된 문서보다 추론이 앞서면 안 된다.


---

## 역할 구조

이 폴더의 역할은 다음과 같이 분리한다.

- `planner/`
- `generator/`
- `tester/`

각 역할은 자기 책임 범위 안에서만 행동해야 하며,  
다른 역할의 책임을 임의로 대신 수행하지 않는다.


---

## 역할 설명

### planner
planner의 역할은 요구사항과 정책을 바탕으로 작업을 분해하고,  
구현 전에 해석 차이를 줄이는 것이다.

planner는 다음을 수행한다.

- 요구사항 분석
- 작업 단위 분해
- 권한/입력값/출력값/예외 상황 정리
- 선행 조건 및 의존성 정리
- 모호한 지점 표시
- 구현 순서 제안

planner는 다음을 하면 안 된다.

- 실제 구현 코드 작성
- 없는 API/DTO/필드 임의 생성
- 요구사항에 없는 기능 제안 후 바로 확정
- 정책이 비어 있는 부분을 자기 판단으로 메우기

planner의 핵심 책임은 **만드는 것**이 아니라 **해석을 줄이는 것**이다.


### generator
generator의 역할은 planner와 shared 문서에서 확정된 내용만 구현하는 것이다.

generator는 다음을 수행한다.

- 확정된 스펙 범위 내 구현
- 변경 파일 및 변경 내용 정리
- 구현된 범위와 미구현 범위 구분
- 정책 위반 없는 코드 작성

generator는 다음을 하면 안 된다.

- shared 문서에 없는 기능 추가
- API 명세에 없는 엔드포인트 임의 생성
- 응답 필드/권한/동작 규칙 임의 변경
- “이 정도는 편의상 필요해 보인다”는 이유로 기능 확장
- planner 없이 요구사항을 자기 식으로 재해석

generator의 핵심 책임은 **창작**이 아니라 **정확한 실행**이다.


### tester
tester의 역할은 구현이 “동작하는가”만 보는 것이 아니라,  
“의도한 요구사항대로 구현되었는가”를 검수하는 것이다.

tester는 다음을 수행한다.

- 요구사항 대비 누락 점검
- 권한 정책 검수
- 응답 구조 검수
- 예외 처리 검수
- 경계값 및 실패 케이스 점검
- generator의 임의 확장 여부 확인

tester는 다음을 하면 안 된다.

- 구현이 되어 있지 않은 기능을 추정으로 통과 처리
- “대충 동작하니 괜찮음”으로 판단
- shared 문서보다 코드의 현재 상태를 우선시
- 명세 누락을 자기 해석으로 보완

tester의 핵심 책임은 **버그 탐지**보다 **요구사항 일치 여부 검증**이다.


---

## 공통 금지 사항

모든 역할은 아래 행동을 금지한다.

- 요구사항에 없는 기능 임의 추가
- 문서에 없는 API/DTO/필드 생성
- 권한을 더 넓게 해석
- soft delete / hard delete 정책 임의 변경
- 댓글/피드백/토글 정책 임의 변경
- 목록 응답 범위 임의 변경
- “프론트에서 처리하면 된다” 또는 “백엔드에서 알아서 가능하다” 식으로 책임 전가
- 명시되지 않은 항목을 일반적인 관례만으로 확정

모호한 부분이 있으면 메꾸지 말고, 모호하다고 표시해야 한다.


---

## 모호한 항목을 만났을 때의 행동 원칙

에이전트는 모호한 항목을 발견했을 때 아래 순서로 판단한다.

1. `shared/` 문서 안에서 해답이 있는지 찾는다.
2. 자신의 역할 문서에 처리 원칙이 있는지 확인한다.
3. 그래도 없으면 **추정하지 말고 미정 사항으로 표시**한다.
4. 미정 사항을 해결하지 못한 채 기능을 확정 구현하지 않는다.

즉, **빈칸을 창의적으로 메우지 않는다.**  
이 프로젝트에서 창의성보다 중요한 것은 일관성과 재현성이다.


---

## 출력 원칙

각 역할은 자신의 결과를 아래 기준으로 정리한다.

- 무엇을 기준으로 판단했는가
- 무엇을 수행했는가
- 무엇은 수행하지 않았는가
- 어떤 문서를 근거로 했는가
- 어떤 항목이 아직 미정인가

결과는 항상 **근거 기반**으로 작성해야 한다.  
“보통은 이렇다”, “관례상 그렇다”는 이유만으로 결정하지 않는다.


---

## 이 폴더의 핵심 원칙 요약

이 `.agent` 폴더의 목적은 에이전트가 똑똑하게 추측하게 만드는 것이 아니다.  
오히려 반대로, **추측하지 못하게 만들기 위한 구조**다.

핵심 원칙은 아래와 같다.

- 공통 기준은 `shared/`에서 먼저 확인한다.
- 역할은 분리하되, 기준은 공유한다.
- planner는 해석을 줄인다.
- generator는 확정된 것만 구현한다.
- tester는 요구사항 일치 여부를 검수한다.
- 비어 있는 부분은 채우지 말고 드러낸다.
