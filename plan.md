# DOGWAJA 로깅 시스템 구축 계획 (Logging Plan)

백엔드(Spring Boot)와 프론트엔드(Next.js)에서 모두 적용되는 전사적 로깅 체계 구축을 위한 계획입니다. **기존 비즈니스 로직을 전혀 변경하지 않고** 요구된 15가지 로깅 원칙을 어떻게 실제 코드에 반영할지 설계했습니다.

## 1. 목적에 따른 로그 분리 및 보관 정책 (원칙 1, 15)
로그의 성격에 따라 파일과 관리 주기를 분리합니다.

- **운영 로그 (application.log):** 시스템 기동, 스케줄링 동작, 일반적인 INFO 레벨 흐름
  - 보관 주기: 30일, 일 단위 롤링 및 압축
- **에러 로그 (error.log):** WARN/ERROR 레벨 전용 로그로 장애 추적에 활용
  - 보관 주기: 90일, 일 단위 롤링 및 압축
- **비즈니스 이벤트 로그 (business.log):** 중요한 이벤트(사용자 로그인/주문/결제 등)를 별도의 채널이나 특정 마커로 기록
- **접근 로그 (access.log):** Nginx 등 웹 서버나 Spring Tomcat의 Access Log

## 2. 의미 있는 내용과 구조화된 포맷 사용 (원칙 2, 6, 13)
단순한 텍스트가 아닌, 검색과 필터링이 용이하도록 JSON 포맷이나 최소한 일관된 형태의 패턴을 적용합니다.

- **포맷 예시:** `[시간] [레벨] [TraceID] [사용자ID] [서비스명] - [요청URL] [이벤트:결과] 상세메시지`
- **나쁜 예:** `로그인 실패`
- **좋은 예:** `{ "level":"ERROR", "userId":123, "event":"login_failed", "reason":"InvalidPassword", "message":"로그인 실패" }` 혹은
  `로그인 실패: userId=123, action=login, reason=InvalidPassword`

## 3. 로그 레벨 기준 확립 (원칙 3, 10, 12)
팀 내 로그 레벨 기준을 명확히 하고, 환경(Profile)별로 레벨을 차등 적용합니다.

| 레벨 | 적용 대상 | 환경별 적용 |
| --- | --- | --- |
| **TRACE** | 프레임워크 내부 동작, 상세 SQL 바인딩 등 | Local/Dev |
| **DEBUG** | 디버깅용 상세 흐름, 외부 API 요청/응답 등 (보안 안전 시) | Local/Dev |
| **INFO** | 주요 비즈니스 흐름(로그인 성공, 스낵 추천 생성 등) | 전체 환경 |
| **WARN** | 클라이언트 입력 오류(4xx 일부), 인증 실패, 권한 부족 등 (장애 아님) | 전체 환경 |
| **ERROR** | 서버 예외(5xx), DB 접속 실패, 시스템 주요 기능 마비 | 전체 환경 |

## 4. 민감 정보 마스킹 정책 (원칙 4)
개인정보 및 인증 정보는 마스킹 처리하여 절대 평문으로 노출시키지 않습니다.

- **절대 기록 금지:** 비밀번호 원문, 액세스/리프레시 토큰, 민감한 개인 식별 고유 정보
- **마스킹 필요 항목:** 전화번호, 이메일 (`010-****-1234`, `t***@gmail.com`)
- **구현 방식:** Logback의 커스텀 포맷터나 Json 변환 시 마스킹 룰 적용

## 5. 예외는 삼키지 않되, 한 번만 기록 (원칙 5)
예외는 발생한 지점에서 바로 로그만 남기고 무시(`catch하고 로그만 출력`)하지 않습니다. 대신 의미 있는 상위 계층으로 던져, `@RestControllerAdvice` 같은 글로벌 영역에서 요청 문맥과 함께 한 번만 명확히 기록합니다.

- **Repository / Service:** Exception을 감지하면 커스텀 예외로 묶어 throw
- **GlobalExceptionHandler:** HTTP 요청 정보, 내부 상세 디테일 수집 후 `ERROR/WARN` 로그 출력 및 사용자용 정제 메시지 반환

## 6. 요청 단위 추적을 위한 Trace ID (원칙 7)
MDC(Mapped Diagnostic Context)를 활용하여 각 요청에 고유 식별자를 부여, 분산 환경이나 계층 간 로그를 결합 분석합니다.

- **구현 방식:** Filter/Interceptor에서 HTTP 요청이 시작될 때 UUID(`traceId`)를 생성해 `MDC.put("traceId", uuid)` 저장
- 로그 포맷(레이아웃)의 최상단에 이 `traceId`를 고정 노출하여 장애 발생 시 하나의 요청 흐름 전체를 추적

## 7. 이벤트 중심 로깅 및 외부 연동 구간 집중 (원칙 8, 9)
메서드 단순 호출 로그보다는 비즈니스 시나리오 관점에서 로그를 남기며, 백엔드 문제의 단골 요인인 '외부 연동' 로깅은 반드시 포함합니다.

- **중요 이벤트:** 회원가입 성공/실패, 간식 요청(주문) 생성, 커멘트 등록 등
- **외부 연동:** DB 연결 및 타 시스템 통신 구간에서는 시스템명, 성공 여부, 소요 시간(ms), 실패 원인을 기록. 단, 대용량 요청 본문은 과도하게 기록하지 않음.

## 8. 모니터링/메트릭 분리 및 성능 영향 최소화 (원칙 11, 14)
로그를 많이 쓴다고 해서 모니터링이 자동화되지는 않습니다.

- 시스템 상태 척도(에러 발생 빈도)는 메트릭/알람의 역할(추후 도입 고려)이며, 에러 로그는 그 "결과에 대한 원인"을 밝히는 역할을 수행.
- 성능 저하를 방지하기 위해 로직 루프 내부에 반복 로그 작성을 지양, 거대한 엔티티 전체 직렬화를 금지.
- Logback의 `AsyncAppender`를 적용하여 I/O 비용이 비즈니스 로직 스레드를 막지 않도록 조치.
