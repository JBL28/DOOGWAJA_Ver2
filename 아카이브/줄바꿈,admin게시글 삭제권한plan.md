# 추천글 기능 버그 수정 및 개선 플랜

> **작성일:** 2026-04-01
> **대상 기능:** 추천 게시글/댓글 삭제 권한, 댓글 작성 UI, 줄바꿈(\n) 보존 및 렌더링 오류 해결

---

## 1. 개요
현재 구현 완료된 추천글 도메인에서 사용자의 피드백에 따라 다음과 같은 세 가지 사항을 수정하고 보완해야 합니다.
1. admin유저의 타인 게시글 삭제 권한 추가
2. 댓글 작성 UI 컴포넌트 추가 및 기능 연동
3. 게시글 및 댓글 줄바꿈 문자가 DB에 누락되는 문제 및 화면 출력(렌더링) 오류 보완

---

## 2. 작업 단위 분해

### 작업 1: 관리자(ADMIN) 삭제 권한 추가
**목표:** 관리자가 타인의 추천 게시글 및 추천 댓글을 삭제할 수 있도록 프론트엔드 및 백엔드 권한 검증 로직을 수정합니다.
**근거 문서:** 사용자의 특별 요구사항 1번 (admin은 타인의 게시글도 삭제 가능해야한다.)
**선행 조건:** 기존 삭제 API와 권한 로직 작동 여부 확인
**작업 범위:**
- 백엔드: 추천 게시글 삭제 로직 (필요 시 댓글 삭제 로직도 포함)
- 프론트엔드: 메인 페이지 내 삭제 버튼 활성화 조건 로직 수정
**제외 범위:** 삭제가 아닌 수정 권한 (ADMIN이 타인의 글을 임의 수정하는 것은 제외, 삭제만 해당함)
**세부 정책:**
- 삭제 API 호출 시 서버측 권한 검증 로직:
  - 기존: `로그인한 유저의 ID == 작성자의 ID` 일 경우에만 허용
  - 변경: `로그인한 유저의 ID == 작성자의 ID` 이거나, 로그인한 유저의 Role(권한)이 `ADMIN` 인 경우에도 삭제 로직 통과 및 수행 반환.
- 프론트엔드 UI 수정: 
  - `page.tsx` 등 리스트에서 본인의 게시물에만 나타나던 "삭제" 버튼을, 접속한 계정이 `ADMIN`인 경우에도 표출되도록 조건식 수정(`user?.role === 'ADMIN'`).
**완료 조건:**
- ADMIN 계정으로 로그인 후 타인의 추천 게시글 삭제 시 200 응답이 오고 정상적으로 삭제된다.
- 일반(USER) 계정으로 타인의 글을 삭제 시 403 예외 응답을 반환한다.
- 메인 페이지에서 ADMIN 사용자는 작성자가 자신이 아님에도 '삭제' 버튼을 확인할 수 있다.
**미정 사항:** 없음.
**generator 전달 메모:** 
- 서비스 메서드 단의 권한 검증 부분(`if (authorId != userId)`)에서 Spring Security Context 정보, 혹은 엔티티의 Role 정보를 불러와 어드민 여부를 분기해야 합니다.
**tester 전달 메모:** 
- 여러 계정(ADMIN 역할, 일반 USER 역할)을 사용하여 삭제 API 권한 분리가 명확한지 테스트해 주세요.

### 작업 2: 메인 페이지 내 댓글 작성 UI 추가
**목표:** 프론트엔드 메인 페이지의 각 추천글 카드 내부에 댓글 입력을 위한 폼을 신설하고 API와 연결합니다.
**근거 문서:** 사용자의 특별 요구사항 2번 (댓글 작성을 위한 UI가 없다. 관련 컴포넌트 추가 등.)
**선행 조건:** 현재 `frontend/app/page.tsx`에서 추천글과 댓글 리스트를 출력하고 있음.
**작업 범위:** 
- 프론트엔드: `frontend/app/page.tsx` 내 추천글 카드 하단에 댓글 폼 UI 추가
- 프론트엔드: `lib/request.ts`의 `postRcComment` 연동 및 상태 변경
**제외 범위:** 대댓글 UI
**세부 정책:**
- 폼 위치: 추천글 카드의 댓글 미리보기(또는 전체 댓글 목록) 근처, 자연스러운 하단부.
- 권한 및 노출: 로그인 상태(`user != null`)에서만 폼을 보여주되, 미로그인 시에는 '로그인 후 댓글을 달 수 있습니다' 같은 구문이나 disabled 처리 활용.
- 입력값 검증: 내용이 빈 상태로 제출을 시도하면 경고(alert) 및 요청 방지.
- 상태 갱신: "등록" 버튼 직후, 새로고침 없이 작성한 댓글이 즉시 화면의 목록에 반영되고 댓글 수 카운트(`commentCount`)도 증가하여야 함 (추천글 목록을 새로 fetch 하거나 로컬 상태를 수동 업데이트).
**완료 조건:**
- 메인 페이지의 각 추천글 카드 하단에 댓글 입력 요소가 존재한다.
- 내용을 입력하고 등록하면 해당 카드의 댓글 목록에 추가된 댓글이 실시간으로 확인된다.
**미정 사항:**
- 상태 업데이트를 낙관적 업데이트(optimistic update)로 할지, 단순히 fetch API(ex: `loadRecommendations`)를 하나 다시 호출할지는 구현체의 재량에 맡김.
**generator 전달 메모:** 
- `page.tsx`가 이미 무거우므로 가능하면 간결한 폼으로 구현해 주세요. 기존 로드된 상태 값(`expandedComments`, `recommendations`)과의 렌더링 동기화가 가장 중요합니다.
**tester 전달 메모:** 
- 로그인, 비로그인 상태 각각 접속해 댓글창 렌더링 및 기능 통제 여부를 점검해 주세요.

### 작업 3: 게시글 줄바꿈(\n) 적용 및 렌더링 문제 해결
**목표:** 줄바꿈 문자가 들어간 텍스트가 정상적으로 DB에 저장되도록 하고 브라우저에도 줄바꿈으로 반영하도록 수정합니다. (현재 프론트에서 \n 보존을 놓치고 있거나 태그 문제로 추정됨)
**근거 문서:** 사용자의 특별 요구사항 3번 (게시글에 줄바꿈이 적용되지 않는 문제. DB 저장 단계부터 공백 등으로 소실됨)
**선행 조건:** 게시글 및 댓글 작성 기능
**작업 범위:**
- 프론트엔드: 글 작성 및 수정 페이지 (`frontend/app/recommendations/new/page.tsx`, `frontend/app/recommendations/[rcId]/edit/page.tsx`), 그리고 이번에 신설되는 '댓글 작성 UI'
- 프론트엔드: 메인 페이지 조회 컴포넌트(`frontend/app/page.tsx`)
**제외 범위:** 마크다운 렌더러 도입 등 외부 라이브러리 사용 배제.
**세부 정책:**
- 입력 폼: 글의 내용(추천 이유)이나 댓글 입력 영역에 `<input type="text">` 태그를 사용 중이라면, 이를 `<textarea>` 요소로 교체하여 엔터 키 입력 시 명확한 개행(`\n`)이 클라이언트에서 캡처되도록 한다.
- 출력(렌더링): `page.tsx` 등 추천 이유 노출부 및 댓글 노출부의 HTML 컨테이너 스타일에 `white-space: pre-wrap;` 속성을 반드시 추가하여 CSS가 개행 문자를 화면 구성에 반영하도록 한다.
- 백엔드 점검(옵션): Controller~Service 구간에서 String의 줄바꿈 문자를 치환/소실하는 어노테이션이나 유틸 로직이 있지 않은지 점검한다. 없다면 프론트엔드의 `textarea` 교체만으로 해결된다.
**완료 조건:**
- 브라우저상에서 글 작성/수정 시 다중 줄바꿈이 허용되며 작성된다.
- 등록 완료 후 DB의 칼럼에 실제 줄바꿈이 존재하는 데이터로 인서트된다.
- 화면 조회(`page.tsx`)에서 줄바꿈이 의도된 그대로 렌더링되어 보인다.
**미정 사항:** 없음.
**generator 전달 메모:** 
- `reason`과 `content`를 렌더링하는 태그 요소(p, span, div)에 스타일로 `white-space: 'pre-wrap'`이 누락되지 않도록 특히 주의해야 합니다.
**tester 전달 메모:** 
- 의도적으로 줄바꿈을 3번 이상 친 문자열로 신규 게시글과 댓글을 등록해 본 뒤 목록 레이아웃을 확인하세요.
